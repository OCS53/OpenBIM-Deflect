# OpenBIM-Deflect — MVP (v1.0) 핵심 스펙

가장 단순하고 **확실하게 한 사이클이 끝까지 도는 뼈대**를 먼저 구축하는 것을 목표로 합니다. 이 문서는 README의 로드맵 항목 *「단순 단일 부재(Beam/Column) 대상 정적 하중 해석 파이프라인」*에 대응하는 구체적 스펙입니다.

---

## 1. 입력 모델 (대상 IFC)

| 항목 | 스펙 |
|------|------|
| **범위** | 단일 부재만 포함한 **아주 단순한** `.ifc` 파일 |
| **예시** | 단순 기둥 하나, 또는 보 하나로만 구성된 모델 |
| **전제** | 복합 구조·다수 부재·복잡 어셈블리는 v1.0 범위 밖 |

> v1.0에서는 파일 복잡도를 제한해, 파이프라인(추출 → 메쉬 → 해석 → 시각화) 검증에 집중합니다.

---

## 2. 프론트엔드 (웹 UI)

### 2.1 파일 입력

- 사용자가 `.ifc` 파일을 브라우저에 **드래그 앤 드롭**으로 업로드합니다.

### 2.2 3D 뷰

- 업로드된 모델의 부재가 **3D 화면**에 표시됩니다. (기하 표현은 MVP 단계에서 단순화 가능)

### 2.3 하중 입력 패널

단순한 UI 패널에서 다음을 입력합니다.

| 입력 | 설명 | 예시 |
|------|------|------|
| **하중 (Load)** | 크기 | 예: `10` kN |
| **방향** | 하중이 가해지는 방향 | 예: **Z축 아래** (전역 좌표계 기준으로 정의) |

> v1.0에서는 지지 조건·재료 등은 **합리적인 기본값** 또는 사전 약속된 단순 규칙으로 두고, UI는 하중 크기·방향 중심으로 시작해도 됩니다. (필요 시 최소 필드만 추가)

---

## 3. 백엔드 (연산 파이프라인)

아래 순서로 **한 파이프라인**으로 연결합니다.

1. **IFC 읽기 및 기하 추출**  
   - **IfcOpenShell**로 업로드된 파일을 읽고, 해석에 사용할 **기하학적 솔리드**(또는 이에 준하는 표면/체적 표현)를 추출합니다.

2. **메쉬 생성**  
   - **Gmsh** 등을 사용해 **3D 유한요소 해석용 메쉬**로 분할합니다.

3. **정적 구조 해석**  
   - 오픈소스 솔버로 **정적 해석(Static Analysis)** 을 수행합니다.  
   - README와의 정합을 위해 **1순위: CalculiX** 를 가정할 수 있으며, 동등한 역할이면 **OpenSees** 등으로 대체 가능함을 스펙에 포함합니다.

4. **결과 집계**  
   - 노드/요소 단위 **변위(처짐)** 및 **응력(또는 응력에 준하는 스칼라 필드)** 를 추출해 API 응답 형태로 정리합니다.

---

## 4. 결과 출력 (프론트 연동)

### 4.1 처짐(Deflection) 수치

- 계산된 **처짐량**을 프론트엔드로 반환합니다.  
  - 예: *중앙부가 약 5 mm 휘어짐* 과 같이, 대표값 또는 요약 지표 + (가능하면) 노드별 변위 필드.

### 4.2 3D 시각화

| 요소 | 스펙 |
|------|------|
| **변형 형상** | 휘어진 모습을 **과장 배율(Scale factor)** 을 적용해 3D 화면에 렌더링합니다. (실제 변위가 작아도 눈에 띄게) |
| **응력 하이라이트** | 응력이 **집중된 구간**을 시각적으로 강조합니다. v1.0에서는 **붉은 색** 등으로 표현하는 것을 목표로 합니다. |

README의 기술 스택과 맞추면, 결과 필드 시각화에는 **VTK.js** 등을 사용하는 방향이 자연스럽습니다.

---

## 5. v1.0 완료 기준 (권장)

다음이 **한 번에** 성립하면 MVP v1.0 뼈대 완료로 봅니다.

1. 단순 단일 부재 IFC 업로드 → 3D 표시  
2. 하중(kN) + 방향(Z 아래 등) 입력 → 백엔드 해석 실행  
3. 처짐(및 응력 필드) 반환 → 변형 과장 표시 + 고응력 구간 붉은 표시  

이후 버전에서 다중 부재, 하중 종류 확장, 재료·경계 UI 고도화 등을 로드맵에 따라 확장합니다.

---

## 6. 구현 전 준비 사항

구현에 들어가기 전에 아래를 맞춰 두면 범위 난립·단위 불일치·병목 발견 시점을 줄일 수 있습니다.

### 6.1 범위와 「완료」 정의 고정

- 본 문서 **§1~§5**의 MVP 범위(단일 부재·정적 하중·최소 UI)를 v1.0의 경계로 고정하고, 부재 선택·복수 부재·풍하중 등은 **명시적으로 제외**합니다.
- 팀 내에서 「성공」을 **한 IFC → 한 번 해석 → 화면에 처짐·응력 표현**까지로 한 줄 정의해 두면 구현 중 판단이 단순해집니다.

### 6.2 검증용 자산 준비

- 파이프라인 디버깅용으로 **아주 단순한 단일 보/기둥 IFC**를 1~2개 확보합니다.
- 저장소에는 코드로 재생성 가능한 샘플을 [`sample/`](../sample/)에 둡니다.
  - `sample/simple_beam.ifc`, `sample/simple_column.ifc`
  - 재생성: `sample/requirements.txt`로 IfcOpenShell 설치 후 `python sample/generate_fixtures.py` ([`sample/README.md`](../sample/README.md) 참고)
- (선택) 이론해·다른 CAE와 비교 가능한 **단순 케이스**를 정해 두면 솔버·메쉬 튜닝에 유리합니다.

### 6.3 좌표·단위·관례 문서화

- IFC **전역 좌표**, 하중 **kN/N**, 길이 **m/mm**, 솔버 입력 **단위 일관**을 한 페이지에 적습니다.
- UI에서 말하는 「Z축 아래」와 솔버·후처리에서의 부호·축 방향이 **같게** 보이도록 팀과 합의합니다.

### 6.4 API·데이터 계약 선행

- 업로드, job 생성, 상태 폴링, 결과(JSON: 변위·응력 또는 요약 + 메쉬 등)를 **OpenAPI 스키마 또는 공유 타입**으로 먼저 고정하면 프론트·백·워커를 병렬로 진행하기 쉽습니다.
- [`PROJECT-STRUCTURE.md`](./PROJECT-STRUCTURE.md)의 API·작업 흐름과 맞춥니다.

### 6.5 기술 스파이크(리스크 실험)

- **IfcOpenShell → (중간 형상) → Gmsh → CalculiX** 를 UI 없이 스크립트/노트북으로 **한 번** 끝까지 돌려 봅니다. 여기서 막히면 웹 레이어만으로는 진행이 어렵습니다.
- **Docker 한 벌**에 CalculiX·Gmsh·Python(IfcOpenShell)을 같이 두고 검증합니다. 저장소에는 [`docker/spike/Dockerfile`](../docker/spike/Dockerfile)과 `docker compose run --rm pipeline-spike`([`scripts/spike/README.md`](../scripts/spike/README.md))로 동일 환경을 재현할 수 있습니다.

### 6.6 실행·배포의 최소선

- `docker-compose`로 API·Redis·worker·(선택) 프론트를 한 번에 띄우는 것을 목표로 하되, 첫 단계에서는 로컬 Python+Redis만으로도 동작하게 나눌지 정합니다.
- `.env.example`, 로그, 작업 디렉터리(`data/jobs/…` 등) 관례를 초기에 맞춥니다.

### 6.7 리포지토리·협업 관습

- 브랜치·PR·커밋 메시지 등 **최소 규칙**을 합의합니다.
- README에 언급된 `CONTRIBUTING.md`는 짧은 형태라도 두면 기여·온보딩에 도움이 됩니다.

### 6.8 라이선스·서드파티

- IfcOpenShell, Gmsh, CalculiX, VTK.js 등 **라이선스 호환**과 배포 시 **고지(NOTICE)** 필요 여부를 한 번 정리합니다.
